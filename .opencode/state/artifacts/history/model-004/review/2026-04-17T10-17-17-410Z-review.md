# Review — MODEL-004: Generate horse-war via Blender-MCP

## Verdict: REJECT

## Summary

All 5 acceptance criteria FAIL. The generated `horse-war.glb` is an empty/invalid file (4124 bytes, 0 triangles) because the implementation never passed `input_blend`/`output_blend` to `scene_batch_edit` calls. Audit log confirms every `scene_batch_edit` job_start record for MODEL-004 shows `input_blend: null` and `output_blend: null`, with warnings: "No output_blend was provided. Scene changes from this job are in-memory only, saved_blend will remain null."

This is the **same root cause** as MODEL-003's initial failures. MODEL-003 resolved only after implementing correct explicit blend chaining with non-null `input_blend`/`output_blend` on every `scene_batch_edit` call.

## Root Cause

**Caller-level defect**: The implementer called `scene_batch_edit` without providing `input_blend` and `output_blend` parameters. The Blender MCP bridge correctly warns on every such call that changes are in-memory only and `saved_blend` will be null.

The bridge itself is working correctly — it properly forwards blend paths when provided (as proven by MODEL-003's successful chain). The defect is in the **caller's workflow**: for MODEL-004, the implementer did not implement the explicit chaining pattern that MODEL-003 eventually used.

---

## Findings and Blockers

### Blocker: BLENDER-MCP-CHAIN (caller-side)

All 5 ACs FAIL because the caller did not pass `input_blend`/`output_blend` to `scene_batch_edit`.

**Resolution path**: Implement the same explicit blend-chaining pattern that MODEL-003 used:
1. `project_initialize(output_blend=stage1.blend)` → creates init scene
2. `scene_batch_edit(input_blend=stage1.blend, output_blend=stage2.blend, operations=[...])` → mutates and saves
3. Chain continues: `scene_batch_edit(input_blend=stage2.blend, output_blend=stage3.blend, ...)`

Every `scene_batch_edit` call in the chain must have explicit non-null `input_blend` and `output_blend`.

### Note on REMED-013

MODEL-004's `follow_up_ticket_ids` includes `REMED-013`. REMED-013 is currently in `planning` stage and addresses the same Blender-MCP chaining contract (EXEC-BLENDER-001). However, the evidence here shows the issue is **not a bridge defect** — the bridge works correctly when called properly. The issue is the caller not using the correct pattern. REMED-013 may need to be reframed as a process/pattern follow-up rather than a bridge repair.

## Next Action

**Return to implementation** with the correct blend chaining pattern:
- `project_initialize` → create named staging blend
- `scene_batch_edit` with explicit `input_blend` and `output_blend` on every call
- Chain the output of each call as input to the next
- Export from the final chained blend file
