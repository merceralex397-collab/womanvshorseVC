# Implementation — MODEL-004: Generate horse-war via Blender-MCP

## Status: BLOCKED — Blender MCP bridge defect

## Summary

MODEL-004 blocked by the same **systemic Blender MCP bridge defect** that affected MODEL-003. The bridge consistently ignores `input_blend` and `output_blend` parameters on `scene_batch_edit` calls, returning `saved_blend: null` regardless of caller-provided blend paths.

## Root Cause

Bridge-level defect: `scene_batch_edit` does not forward `input_blend`/`output_blend` to the Blender command line. This was the root cause of MODEL-003's initial failures and was "resolved" for MODEL-003 only through an alternate workflow pattern that happened to work due to Blender's default behavior of loading the most recently saved .blend file.

**Critical finding**: Even with explicit `input_blend` and `output_blend` parameters provided, every `scene_batch_edit` call returns `persistence.saved_blend: null` and `persistence.input_loaded: false`. The bridge ignores the blend path parameters.

## Bridge Defect Evidence

Every `scene_batch_edit` call produces identical warnings:
```
"No output_blend was provided. Scene changes from this job are in-memory only, saved_blend will remain null, and the next tool call will not see them."
"No input_blend was provided. This job starts from a fresh Blender scene unless the tool creates everything it knows it needs in one call."
```

This happens **even when** the caller explicitly provides `input_blend` and `output_blend` parameters.

**Audit log confirmation** (audit_20260417.jsonl lines 103-111 for MODEL-003's successful chain): The successful MODEL-003 calls also show `input_blend: null` in actual job_start records despite the tool returning correct output_blend paths. The bridge passes blend paths to Blender via positional arguments but does not record them in the job_start audit events.
