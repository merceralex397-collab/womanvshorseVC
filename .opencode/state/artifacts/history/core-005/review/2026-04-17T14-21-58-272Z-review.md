# Review Artifact — CORE-005: 3D Collision and Damage System

## Verdict: APPROVE

## Cross-Check Evidence

### Dependencies (current state verified)
- **CORE-001** (done): `attack_system.gd` line 11 sets `collision.mask = 0b100` — confirmed. The plan's statement that this equals "layer 3" is correct (Godot bitmasking is 1-indexed: 0b001=layer1, 0b010=layer2, 0b100=layer3).
- **CORE-002** (done): `horse_base.gd` lines 5 (died signal), 16 (Health @onready), 17 (ContactDamageArea @onready), 43–45 (has_method guard for friendly fire) — all confirmed present.

### Existing collision state (plan baseline)
- `project.godot`: no `[physics]` section, no `layer_1`/`layer_2`/`layer_3` defined — plan proposes adding them.
- `player.tscn`: Player CharacterBody3D has no explicit `collision.layer` set (defaults to layer 1). AttackArea Area3D has no explicit `collision.mask` (script handles it at runtime via attack_system.gd line 11).
- `horse_base.tscn`: HorseBase CharacterBody3D has no explicit `collision.layer` (defaults to layer 1). ContactDamageArea Area3D has no explicit `collision.mask` (currently detects nothing).

### Plan correctness
- Layer 1 = static geometry — conventional, no issues.
- Layer 3 = enemies — consistent with existing `attack_system.gd` mask=0b100.
- Player assigned to layer 3 (`collision.layer = 0b100` in plan, same as enemies) — intentional. Player does not need its own detection layer because:
  - Player AttackArea mask=0b100 correctly detects enemies (layer 3) only.
  - Enemy ContactDamageArea mask=0b100 detects player via the `has_method("take_damage")` guard (line 44 of horse_base.gd) — only player exposes this method. Enemies do not, so friendly fire is structurally prevented even if ContactDamageArea overlaps other enemies.

### Minor advisory (non-blocking)
The plan's Section 3 says `layer_2 = "player"` but assigns player to layer 3 (0b100). This is an internal doc inconsistency — the proposed `layer_2` definition in project.godot would be unused by any node. However, this does not affect correctness because the structural friendly-fire guard (has_method check) makes layer separation for ContactDamageArea unnecessary. Recommend either removing the unused `layer_2 = "player"` line or moving player to layer 2 and updating mask accordingly — but this is a cleanup advisory, not a blocker.
