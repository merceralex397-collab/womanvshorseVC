## Specific Findings

### Blocking Finding: BLENDER-MCP-CHAIN (Systemic)

**Root cause:** The Blender MCP bridge does not forward `input_blend`/`output_blend` parameters from the JSON payload to the Blender command line for `scene_batch_edit`. Every call runs with `--factory-startup`, starting from a fresh Blender scene, making scene chaining impossible.

**Evidence from implementation artifact:**
- Bridge response warnings on every `scene_batch_edit` call:
  ```
  "No output_blend was provided. Scene changes from this job are in-memory only, saved_blend will remain null..."
  "No input_blend was provided. This job starts from a fresh Blender scene..."
  ```
- Persistence state on every call: `input_loaded: false`, `saved_blend: null`, `persisted_for_next_call: false`
- Audit log evidence (`.blender-mcp/audit/audit_20260417.jsonl`) for three separate calls:
  - `20260417T041500Z-a8faea0c7e` — `input_blend: null`, `output_blend: null`
  - `20260417T041506Z-3e594c3555` — `input_blend: null`, `output_blend: null`
  - `20260417T041520Z-07feba12dd` — `input_blend: null`, `output_blend: null`

**Same root cause as:**
- MODEL-003 original implementation (superseded `2026-04-17T03:35:22-518Z`)
- MODEL-003 plan rollback (superseded `2026-04-17T03:44:07-881Z`)
- MODEL-004 review REJECT (`2026-04-11T16-22:46-995Z`)

**Remediation status:** REMED-009 demonstrated correct chaining works when the caller explicitly passes blend paths and the bridge receives them. However, the actual `scene_batch_edit` tool call path still shows null blend paths in audit logs — indicating either the bridge requires repair, or a pipeline script must implement the explicit chaining loop, or AC reframing is needed. No in-ticket repair path exists without external bridge repair or pipeline script development.
