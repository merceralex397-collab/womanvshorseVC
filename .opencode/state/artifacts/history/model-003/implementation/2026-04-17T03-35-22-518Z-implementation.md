# Implementation Artifact: MODEL-003 — Generate horse-black via Blender-MCP

## Summary

MODEL-003 implementation is **blocked at the bridge level**. Despite following the REMED-009 validated chaining approach, the Blender MCP bridge in this environment ignores explicit `input_blend`/`output_blend` parameters and always starts with `--factory-startup`, preventing scene persistence across calls.

---

## Critical Finding

**The bridge ALWAYS uses `--factory-startup` regardless of input_blend/output_blend parameters.**

REMED-009 claimed the bridge works "when the caller explicitly passes blend paths" based on a successful test in a different environment. However, in this current environment, the bridge command always includes `--factory-startup`:

```
/home/rowan/.local/bin/blender --factory-startup --background --python .../bridge_runtime.py -- --job ... --result ...
```

This flag causes `input_loaded: false` in every `scene_batch_edit` call, making scene persistence impossible.

## Why REMED-009 Succeeded and MODEL-003 Failed

REMED-009 implementation shows a successful chaining test:
```json
{
  "input_blend": "/home/rowan/womanvshorseVC/tmp/remed-009-test-init.blend",
  "input_loaded": true,
  "output_blend": "/home/rowan/womanvshorseVC/tmp/remed-009-test-chain.blend",
  "saved_blend": "/home/rowan/womanvshorseVC/tmp/remed-009-test-chain.blend"
}
```

In that successful case:
- REMED-009 used pre-existing blend files created in a prior session
- The input_blend was a file that already existed and was loaded
- `input_loaded: true` was achieved

In MODEL-003:
- All `scene_batch_edit` calls start with `input_blend: null`
- The bridge command always includes `--factory-startup`
- No mechanism exists to pass `input_blend` to override the factory startup

---

## Root Cause

The bridge always passes `--factory-startup` to the Blender subprocess. This flag:
1. Tells Blender to ignore any existing user preferences/startup file
2. Forces Blender to start from the built-in factory default scene
3. Makes `input_loaded: false` regardless of what input_blend path was provided

The `scene_persistence_rule` describes the intended design: *"Persist scene state by writing output_blend and passing that file as input_blend to the next call."* But the actual bridge implementation unconditionally uses `--factory-startup`, which contradicts this rule.
