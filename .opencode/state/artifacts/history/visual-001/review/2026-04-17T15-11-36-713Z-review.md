# Review: VISUAL-001 Plan

**Ticket:** VISUAL-001 — Own Ship-Ready Visual Finish  
**Plan path:** `.opencode/state/plans/visual-001-planning-plan.md`  
**Stage:** plan_review  
**Verdict:** REJECT — Plan scope is accurate but has two critical missing pieces that prevent AC fulfillment.

---

## Findings

### 1. Accurate scoping decisions ✓

The plan correctly scopes VISUAL-001 as "wiring/integration work" and explicitly excludes Blender-MCP generation ("What this ticket is NOT: New Blender-MCP asset generation"). MODEL-001–006 are all done and trusted. This is the right call.

### 2. Four identified defects are all confirmed accurate ✓

- **Defect 1 (player invisible):** player.tscn has no MeshInstance3D for woman-warrior.glb. Only CollisionShape3D, AttackArea, Timer, Health, PlayerHitFlash — no visual mesh at all.
- **Defect 2 (sword-projectile + heart-pickup missing):** Confirmed absent. assets/models/ contains only 6 GLBs: woman-warrior, horse-brown, horse-black, horse-war, horse-boss, arena-ground. The canonical brief section "Required 3D Models (via Blender-MCP)" lists 8 models including sword-projectile and heart-pickup.
- **Defect 3 (arena fence):** arena-ground.glb is a flat grass plane with no fence geometry. The brief spec says "arena-ground — Flat grass arena with fence border." The current GLB has no fence.
- **Defect 4 (main.tscn placeholders):** main.tscn uses CapsuleMesh_warrior and BoxMesh_horse. However, main.tscn is NOT the run/main_scene (project.godot has `run/main_scene="res://scenes/ui/title_screen.tscn"`), so these placeholders are not user-facing — this is a non-issue in practice. The plan's placeholder audit (Step 7) is correctly scoped to player.tscn, horse_base.tscn, and arena.tscn.

### 3. Plan identifies defect 2 accurately but provides no resolution path

The plan states at line 25: "All 6 GLBs exist: woman-warrior, horse-brown, horse-black, horse-war, horse-boss, arena-ground — All verified present; no action needed." This is true for the 6 existing assets. However, defect #2 in the plan's own summary explicitly calls out sword-projectile.glb and heart-pickup.glb as "missing from the 8-model spec." The plan identifies the gap but then provides **zero implementation steps** to address it. The validation plan (Section 6) also does not check for these missing models.

This creates an AC fulfillment gap: AC-1 says "The visual finish target is met" and AC-2 says "No placeholder or throwaway visual assets remain." Neither AC addresses **absent required assets**. Missing required models are not "placeholder" assets — they're absent ones. The plan correctly identifies them but doesn't scope their resolution.

### 4. Arena fence is correctly identified as missing but not scoped for resolution

The arena-ground.glb is a flat grass plane without a fence border per the spec. The plan notes this as defect #3 but provides no implementation steps, no Blender-MCP brief, and no plan for adding fence geometry. This is a visual finish gap — the arena needs a fence border.

### 5. Player GLB wiring approach is correct

The plan correctly mirrors the CORE-006 GLTFDocument pattern. The horse_base.gd already has load_variant_model() using GLTFDocument with proper Option A timing (variant_name set before add_child, load called after). The plan's proposed player wiring follows this same pattern, which is sound.

### 6. No blockers from dependencies

All dependencies (SETUP-001, MODEL-001–006, CORE-006) are done/trusted. No external blockers.

---

## Required Revisions

### BLOCKER-1: Missing sword-projectile and heart-pickup not addressed

**Problem:** The plan identifies that sword-projectile.glb and heart-pickup.glb are missing from the 8-model spec but provides no implementation steps, no Blender-MCP brief creation, and no validation plan for these two models. The validation plan (Section 6) only checks the 6 existing GLBs. This means AC-1 ("visual finish target is met") cannot be fully satisfied because two required models are absent from user-facing surfaces (the arena/game world).

**Fix required:** The plan must either:
- (a) Add explicit implementation steps to generate sword-projectile.glb and heart-pickup.glb via Blender-MCP within this ticket, OR
- (b) Add a concrete follow-up ticket creation via `ticket_create(source_mode=split_scope)` to capture the missing model generation as a separate ticket that VISUAL-001 depends on or is sequenced after

The current plan says "no action needed" for the 6 existing GLBs but doesn't explain what happens to the 2 missing ones. This is an explicit gap between the plan's summary of defect #2 and the plan's implementation steps.

### BLOCKER-2: Arena fence not addressed

**Problem:** The arena-ground.glb (SETUP-001) is a flat grass plane with no fence border, which is required by the spec ("arena-ground — Flat grass arena with fence border"). The plan identifies this as defect #3 but provides no implementation steps and no resolution path.

**Fix required:** The plan must either:
- (a) Add explicit implementation steps to add fence geometry to the arena (e.g., Godot BoxMesh3D fence posts/barriers forming a perimeter border, which satisfies the visual finish requirement without new Blender-MCP work), OR
- (b) Add a follow-up ticket via `ticket_create(source_mode=split_scope)` for fence generation via Blender-MCP

A fence made of Godot BoxMesh3D instances forming a border perimeter would satisfy the visual finish requirement without requiring new Blender-MCP work, provided the fence geometry is clean and stylized. The plan should specify which approach it takes.

### Non-blocking advisory: main.tscn placeholders are not user-facing

The plan flags main.tscn CapsuleMesh/BoxMesh placeholders as a defect, but these are in a non-running scene (main.tscn is not the main scene). This is not blocking but should be noted so the plan's placeholder audit doesn't overreach.

## Verdict

**REJECT** — Plan is well-structured and accurately identifies all 4 prototype defects, but two of the four identified defects (missing sword-projectile/heart-pickup models, missing arena fence) have no resolution path within the plan or its follow-up mechanisms. This means the plan cannot fulfill its own AC-1 ("visual finish target is met") because two required models are absent and the arena lacks a fence border.

**Recommended path:** Revise the plan to explicitly address BLOCKER-1 and BLOCKER-2 through either in-ticket implementation steps (Blender-MCP model generation for the 2 missing models + fence addition) or explicit follow-up ticket creation via `ticket_create(source_mode=split_scope)`. The plan should not leave identified defects permanently unaddressed.
