## Root Cause Analysis

**Same v1 compat bug as MODEL-007 (record "20260418T011712Z-ed2159a069" and "20260418T011716Z-79d51f4fb8")**

The `_wrap_v1_scene_batch_edit` function in `bridge_runtime.py` does not forward `input_blend` and `output_blend` parameters to the Blender `--background` command line. This causes:
1. Each `scene_batch_edit` call starts from a fresh factory-startup Blender (`--factory-startup` always used)
2. The scene from `project_initialize` is never loaded
3. No scene state persists between calls

**Audit evidence:**
- Record 18 (project_initialize): `input_blend: null, output_blend: "/home/rowan/womanvshorseVC/tmp/heart-pickup-stage.blend"` ‚úÖ ‚Äî blend written correctly
- Record 21 (scene_batch_edit #1): `input_blend: null, output_blend: null` ‚ùå ‚Äî bridge dropped blend path forwarding
- Record 23 (scene_batch_edit #2): `input_blend: null, output_blend: null` ‚ùå ‚Äî same defect repeated

## Workaround Attempted
Explicitly passing `input_blend` and `output_blend` in the tool call does not help ‚Äî the bridge silently drops them. This matches the pattern documented in MODEL-007's blocker artifacts.

The fix would need to be in `_wrap_v1_scene_batch_edit` (or equivalent) in:
