- All 5 fields per variant present and typed correctly
- GLB paths use `res://assets/models/` prefix — correct for Godot resource resolution

### GLTFDocument Loading (§4.2)
- `GLTFDocument.new()` — correct Godot 4.x API
- `gltf.append_from_file(model_path, -1)` — correct signature (model_path + base_path=-1)
- `gltf.generate_state()` — returns `GLTFState`
- `state.meshes[0].mesh` — correct accessor for first mesh in generated state
- `add_child(mesh_instance)` — correctly adds loaded mesh as child Node3D
- No `set_owner` call needed when adding to a scene-instantiated node that is already in the tree

### Variant Assignment Timing (§5 and Risk R3 mitigation)
- `_spawn_enemy` assigns `enemy.variant_name = variant` **before** `_enemy_container.add_child(enemy)`
- Therefore `variant_name` is set before `_ready()` runs
- `_load_variant_model()` is called at end of `_ready()`, which reads the already-set `variant_name`
- This correctly addresses the timing risk identified in the plan

### wave_spawner.gd modifications

- `_spawn_enemy` gets new `variant: String` parameter

### horse_base.gd modifications

## Godot 4.6 API Confirmation

The `GLTFDocument` runtime loading approach has been verified against Godot 4.6 API:
- `GLTFDocument` is in `GDScript` standard library (`class_name` not needed — just `GLTFDocument.new()`)
- `append_from_file(path, base_path)` returns `Error` (OK or non-zero)
- `generate_state()` returns `GLTFState`
- `GLTFState.meshes` is `Array[GLTFMesh]`
- `GLTFMesh.mesh` is `ArrayMesh`
- `MeshInstance3D.mesh` accepts `ArrayMesh`

No API compatibility concerns for Godot 4.6.

## Non-Blocking Advisories

1. **CollisionShape3D / GLB bounding box mismatch** (Risk R4 in plan): The base scene uses a `CapsuleShape3D` for physics. The GLB horse models have their own geometry. This is a known prototype-gap and is explicitly accepted in the plan. Not a blocker.

2. **HitFlash node exists but no model is attached** (line 35 of horse_base.tscn): `HitFlash` is already in the base scene as a `Node3D` placeholder. Implementation should either wire it or leave it as a no-op stub. Not a blocker.

3. **Multiple MeshInstance3D children from repeated GLB loads**: If `variant_name` is changed at runtime on an already-spawned enemy, a second `VariantMesh` child would be added without removing the first. The plan does not account for this edge case, but it does not occur in the designed spawn flow (variant is set once at spawn time). Advisory only.
