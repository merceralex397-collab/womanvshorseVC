# Review: CORE-006 — Enemy variants with model swapping

## Verdict: REVISE

## Summary
Plan scope, AC coverage, GLB paths, dependency state, and API design are all sound. However, a **blocking timing defect** in the `_ready()` GLB loading sequence prevents correct operation: `_load_variant_model()` is called during `_ready()`, but `wave_spawner.gd` sets `variant_name` after `add_child`. Every enemy will load the brown GLB regardless of variant assignment.

## 2. Design Soundness — PARTIAL

### 2a. GLTFDocument API ✓
`GLTFDocument.append_from_file` + `generate_state().meshes[0].mesh` is the correct Godot 4 pattern for runtime GLB mesh extraction. API is verified available in Godot 4.6.

### 2b. Skeleton root handling ✓
Plan correctly extracts `state.meshes[0].mesh` and assigns to a fresh `MeshInstance3D`, bypassing the GLB skeleton root entirely. This is the correct approach.

### 2c. Blocking defect — `_ready()` runs before `variant_name` is set

**Sequence in plan's `_spawn_enemy`**:
```
var enemy: Node3D = enemy_scene.instantiate()
_enemy_container.add_child(enemy)   ← _ready() fires here → _load_variant_model() sees variant_name="brown"
enemy.variant_name = variant       ← variant_name is set AFTER add_child
```

**Result**: All enemies load brown GLB regardless of their variant.

The plan's own risk register (row 3, "horse_base `_ready()` runs before wave_spawner sets `variant_name`") acknowledges this as a **Medium** likelihood risk, but provides **no concrete mitigation**. The plan cannot be approved until this is resolved.

**Required fix (one of)**:
- **Option A** (preferred): Call `_load_variant_model()` via `call_deferred` inside `horse_base.gd`, or split GLB loading into an explicit `load_variant_model()` method called by wave_spawner AFTER setting all properties
- **Option B**: Add a `defer_model_load: bool = true` export; `_load_variant_model()` checks this flag and returns early in `_ready()` when true, with wave_spawner calling it after all properties are set
- **Option C**: Have `_load_variant_model()` use a one-shot `await Engine.get_main_loop().process` to defer mesh loading to the next frame, allowing wave_spawner to set `variant_name` first

### 2d. wave_spawner modifications are additive
The plan's `wave_configs` with `variant` field and the `_spawn_enemy(variant: String)` signature do not exist in the current `wave_spawner.gd`. These are additive changes, which is correct. Implementation must add `variant` to each `wave_configs` entry per the plan's table.

### 2e. CollisionShape3D mismatch risk — acceptable ✓
Risk register correctly flags visual/physics bounding-box mismatch. CapsuleShape3D remains; GLB is visual-only. Correctly scoped as prototype-acceptable with no proposed fix needed at this stage.

## 5. Blocking Issues

### BLOCKER-1: `_ready()` timing — variant_name not set when GLB loads
**Severity**: Blocking — the core variant system feature will not work correctly.
**Root cause**: `_load_variant_model()` is called in `_ready()`, but `wave_spawner.gd` sets `variant_name` after `add_child`.
**Effect**: All enemies load brown GLB regardless of their assigned variant.
**Required action**: Plan must specify a deferred loading mechanism (call_deferred, explicit post-property-setup call, or await next frame) so GLB loading occurs after `variant_name` is assigned.

### Advisory notes (non-blocking, for implementer)
1. **Threaded loading**: `GLTFDocument.append_from_file` is synchronous. Boss (wave 5) spawning 1-2 instances is fine, but if enemy count increases, consider `Thread` for GLB loading to avoid frame hitches.
2. **Mesh disposal**: No `mesh.asset_path` clearing is specified; if enemies respawn repeatedly the GLB meshes may accumulate in memory. Not a blocker for current wave 3-5 enemy counts but monitor post-implementation.
3. **wave_spawner `variant` field must be added to ALL 5 `wave_configs` entries** — current code has no `variant` key in any entry. Implementation step 3 in the plan correctly covers this.
